## 인덱스 기반 정렬을 활용한 드라이빙 테이블 조인 최적화 기법

데이터베이스 성능 튜닝에서 조인(Join)은 가장 빈번하게 병목 현상이 발생하는 지점입니다. 특히 대량의 데이터를 조회하면서 정렬(ORDER BY)과 페이징(LIMIT)을 동시에 수행할 때, 실행 계획을 올바르게 이해하지 못하면 시스템 리소스가 급격히 소모되는 상황에 직면하게 됩니다.

이 글에서는 드라이빙 테이블과 드리븐 테이블의 관계를 분석하고, 정렬 기준 컬럼의 위치가 쿼리 성능에 어떠한 극적인 차이를 만드는지 상세히 설명합니다.

---

## 드라이빙 테이블과 정렬의 메커니즘

데이터베이스 옵티마이저는 조인을 수행할 때 가장 먼저 접근할 테이블인 드라이빙(Driving) 테이블과, 그 결과를 바탕으로 데이터를 찾아갈 드리븐(Driven) 테이블을 결정합니다.

### 1. 병목의 원인 분석: 드리븐 테이블 기준 정렬

일반적인 개발 환경에서 드라이빙 테이블은 인덱스 필터링 효율이 좋은 테이블이 선택됩니다. 그러나 정렬 조건이 드리븐 테이블의 컬럼에 걸려 있다면 성능 저하가 발생합니다.

* 문제 상황: 하위 테이블(Sub Table)을 드라이빙으로 선택하여 데이터를 추출했지만, 정렬은 상위 테이블(Main Table)의 특정 컬럼을 기준으로 수행하는 경우입니다.
* 실행 과정: 옵티마이저는 조인 결과 전체를 메모리나 디스크의 임시 테이블에 저장합니다. 이후 전체 데이터에 대해 정렬(Filesort) 연산을 수행하고 나서야 LIMIT을 적용합니다.



| 항목 | 개선 전 상세 내용 |
| :--- | :--- |
| 조인 방식 | Nested Loop Inner Join |
| 드라이빙 테이블 | Sub_Table (조건 인덱스 사용) |
| 드리븐 테이블 | Main_Table (Primary Key 사용) |
| 병목 지점 | 드리븐 테이블 컬럼 기준 정렬 (ORDER BY) |
| 처리 로직 | 임시 테이블 생성 후 전체 데이터 정렬 |

### 2. 성능 최적화의 핵심: 드라이빙 테이블 기준 정렬

성능을 개선하기 위해서는 정렬 기준을 드라이빙 테이블의 인덱스 컬럼으로 변경해야 합니다. 이를 통해 데이터베이스는 조인 도중 즉시 스캔을 중단하는 조기 종료(Early Termination)를 수행할 수 있습니다.

```sql
-- 개선 전: 드리븐 테이블(Main_Table) 컬럼으로 정렬
SELECT *
FROM Sub_Table s
JOIN Main_Table m ON s.main_id = m.id
WHERE s.category = 'PROMOTION'
ORDER BY m.updated_at DESC
LIMIT 1000;

-- 개선 후: 드라이빙 테이블(Sub_Table) 인덱스 컬럼으로 정렬
SELECT *
FROM Sub_Table s
JOIN Main_Table m ON s.main_id = m.id
WHERE s.category = 'PROMOTION'
ORDER BY s.id DESC
LIMIT 1000;
```

개선 후의 쿼리는 다음과 같은 원리로 동작합니다.

* 인덱스 역순 스캔: 드라이빙 테이블의 인덱스를 역순으로 읽으며 이미 정렬된 데이터를 가져옵니다.
* 정렬 작업 생략: 이미 인덱스에 의해 정렬되어 있으므로 추가적인 정렬 연산이 필요하지 않습니다.
* 조기 중단: LIMIT 조건이 조인 중간에 적용되어, 필요한 행 수만큼만 읽고 즉시 스캔을 중단합니다.

### 3. 비교 분석을 통한 최적화 확인

두 방식의 차이는 자원 소모와 실행 효율 측면에서 확연하게 나타납니다.

| 분석 지표 | 개선 전 (드리븐 정렬) | 개선 후 (드라이빙 정렬) |
| :--- | :--- | :--- |
| 전체 실행 시간 | 상대적으로 느림 (병목 발생) | 매우 빠름 (최적화 완료) |
| 스캔 행 수 | 전체 대상 행 스캔 | LIMIT 도달 시 스캔 중단 |
| 임시 테이블 생성 | 조인 결과 저장을 위해 대량 발생 | 발생하지 않음 |
| 정렬 알고리즘 | 후속 정렬 (Filesort) 필요 | 인덱스 스캔으로 정렬 대체 |
| LIMIT 적용 시점 | 정렬이 모두 끝난 후 적용 | 스캔 도중 적용 (Early Stop) |

## 결론 및 활용 방안

조인 쿼리에서 ORDER BY와 LIMIT을 최적화하는 핵심은 정렬 기준을 드라이빙 테이블의 인덱스 컬럼으로 일치시키는 것에 있습니다.

* 핵심 요약: 드리븐 테이블 컬럼으로 정렬하면 조인 결과 전체를 읽어 임시 테이블을 만들어야 하므로 I/O 부하가 일어납니다. 반면 드라이빙 테이블 인덱스로 정렬하면 필요한 데이터 수만큼만 조인을 수행하고 즉시 결과를 반환합니다.
* 실무 적용: 복잡한 조인이 포함된 서비스의 응답 속도가 느리다면 실행 계획을 통해 임시 테이블과 정렬 연산이 발생하는지 확인합니다. 만약 발생하고 있다면 정렬 기준이 되는 테이블이 드라이빙 테이블인지 검토하여 쿼리 구조를 조정해야 합니다.
