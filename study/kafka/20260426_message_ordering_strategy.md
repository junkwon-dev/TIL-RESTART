## 카프카 파티션 확장 시 순서 보장 전략

분산 메시징 시스템인 아파치 카프카(Apache Kafka)를 운영하다 보면 처리량 증대를 위해 파티션(Partition) 수를 늘려야 하는 상황에 직면합니다. 하지만 카프카의 메시지 순서 보장은 동일 파티션 내에서만 유효하기 때문에, 파티션 수를 변경하는 순간 기존의 순서 보장 규칙이 깨지는 문제가 발생합니다.

이 글에서는 파티션 확장 시 발생하는 순서 역전 현상의 원인을 분석하고, 비즈니스 로직의 무결성을 유지하며 파티션을 확장하는 실무적인 해결 방안을 제시합니다.

---

## 파티션 확장과 해시 키의 상관관계

카프카 프로듀서는 메시지를 보낼 때 메시지 키(Key)를 해시 함수에 대입하여 전송할 파티션을 결정합니다. 일반적으로 사용하는 방식은 $hash(key) \pmod{partition\_count}$ 공식입니다.

### 순서가 보장되지 않는 이유

파티션 수가 변경되면 분모에 해당하는 $partition\_count$가 달라집니다. 결과적으로 동일한 키를 가진 메시지라도 확장 전과 후가 서로 다른 파티션으로 할당되는 결과가 나타납니다.



예를 들어 파티션이 2개일 때 키 A가 0번 파티션으로 갔다면, 파티션이 4개로 늘어난 후에는 2번 파티션으로 배정될 수 있습니다. 이때 컨슈머가 0번 파티션의 이전 데이터를 다 읽기 전에 2번 파티션의 새 데이터를 먼저 읽게 되면 데이터 처리 순서가 뒤섞이게 됩니다.

### 파티션 확장 전후 비교

| 항목 | 파티션 확장 전 | 파티션 확장 후 |
| :--- | :--- | :--- |
| 결정 로직 | hash(Key) % N | hash(Key) % (N + M) |
| 메시지 적재 | 기존 파티션 세트에만 적재 | 신규 파티션을 포함하여 분산 적재 |
| 순서 보장 범위 | 단일 파티션 내 보장 | 확장 시점 전후로 키-파티션 매핑 변화 발생 |

## 순서 보장을 위한 단계적 확장 전략

파티션 수를 늘리면서도 데이터의 순서를 엄격하게 지켜야 한다면 다음과 같은 세 가지 전략을 고려할 수 있습니다.

### 1. 커스텀 파티셔너(Custom Partitioner) 활용

기본 해시 전략 대신, 특정 기간 동안 기존 키들이 기존 파티션 번호를 유지하도록 로직을 작성합니다.

```java
public class OrderPreservingPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes, 
                         Object value, byte[] valueBytes, Cluster cluster) {
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        
        // 기존 파티션 개수가 2개였다고 가정
        int oldPartitionCount = 2;
        
        // 특정 로직이나 타임스탬프를 기준으로 구형 키는 기존 파티션 범위 내에서 할당
        return Utils.toPositive(Utils.murmur2(keyBytes)) % oldPartitionCount;
    }
}
```

이 방식은 단기적으로 순서를 보장하지만, 신규 파티션의 활용도를 떨어뜨려 부하 분산의 이점이 줄어든다는 단점이 있습니다.

### 2. 신규 토픽 생성 및 마이그레이션 (추천 방식)

가장 안전하고 확실한 방법은 원하는 파티션 수를 가진 새로운 토픽을 생성하는 것입니다.



* Step 1: 신규 파티션 개수를 가진 토픽 B를 생성합니다.
* Step 2: 프로듀서가 신규 데이터를 토픽 B로 전송하도록 변경합니다.
* Step 3: 컨슈머는 기존 토픽 A의 데이터를 모두 소비(Drain)할 때까지 기다립니다.
* Step 4: 토픽 A의 소비가 완료되면 컨슈머가 토픽 B를 바라보게 합니다.

### 3. 애플리케이션 레벨의 순서 재정렬

데이터 내부에 시퀀스 번호나 타임스탬프를 포함합니다. 컨슈머가 데이터를 읽은 후, 로컬 버퍼나 상태 저장소(State Store)에서 순서에 맞게 재정렬한 뒤 로직을 수행합니다. 이는 카프카의 파티션 구조에 의존하지 않는 가장 유연한 방식입니다.

## 결론 및 제언

카프카에서 파티션 확장은 단순한 설정 변경 이상의 의미를 갖습니다. 특히 주문 처리나 잔액 계산처럼 순서가 생명인 시스템에서는 파티션 수 변경이 데이터 정합성 오류로 이어질 수 있습니다.

### 핵심 요약
1. 파티션 수 변경은 해시 결과값을 변화시켜 순서 보장을 깨뜨립니다.
2. 실무에서는 가급적 신규 토픽으로의 마이그레이션을 권장합니다.
3. 불가피한 경우 커스텀 파티셔너를 통해 확장 과도기를 관리해야 합니다.
