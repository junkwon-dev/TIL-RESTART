## 개요

애플리케이션의 성능 최적화를 논할 때 캐싱(Caching)은 빼놓을 수 없는 핵심 요소입니다. 반복되는 데이터베이스 조회나 복잡한 연산 결과를 메모리에 저장하여 응답 속도를 획기적으로 개선할 수 있기 때문입니다.

하지만 비즈니스 로직 사이에 캐시 로직이 섞이게 되면 코드의 가독성이 떨어지고 유지보수가 어려워지는 문제가 발생합니다. Spring 프레임워크는 이러한 문제를 해결하기 위해 Spring Cache Abstraction(캐시 추상화)을 제공합니다.

이 기능을 활용하면 개발자는 특정 캐시 저장소 기술에 종속되지 않고, 어노테이션 기반의 설정만으로 효율적인 캐싱 아키텍처를 구축할 수 있습니다. 본 글에서는 Spring Cache의 핵심 개념과 작동 원리, 그리고 실무 적용 시 주의사항을 심도 있게 다룹니다.

## Spring Cache의 핵심 개념과 원리

### 1. 캐시 추상화와 AOP

Spring Cache의 가장 큰 특징은 추상화에 있습니다. 마치 JDBC가 다양한 데이터베이스를 동일한 인터페이스로 다루듯, Spring Cache는 캐시 저장소의 종류(Redis, Ehcache, Caffeine 등)와 관계없이 동일한 방식으로 캐시를 적용할 수 있게 합니다.

작동 원리의 핵심은 AOP(Aspect Oriented Programming)입니다. @Cacheable과 같은 어노테이션이 붙은 메서드가 호출되면 Spring은 해당 빈의 프록시(Proxy) 객체를 생성하여 호출을 가로챕니다.



| 단계 | 수행 동작 | 설명 |
| :--- | :--- | :--- |
| 1 | 메서드 호출 가로채기 | 사용자가 메서드를 호출하면 프록시 객체가 개입합니다. |
| 2 | 캐시 조회 | Cache Manager를 통해 설정된 Key에 해당하는 데이터가 있는지 확인합니다. |
| 3 | 분기 처리 | 데이터가 존재하면(Cache Hit) 메서드를 실행하지 않고 즉시 반환합니다. |
| 4 | 메서드 실행 | 데이터가 없으면(Cache Miss) 실제 로직을 수행한 후 결과를 캐시에 저장합니다. |

---

### 2. 핵심 어노테이션 (The Big 3)

Spring에서 캐시를 제어하기 위해 가장 빈번하게 사용하는 세 가지 어노테이션의 특성은 다음과 같습니다.

#### @Cacheable: 데이터 저장 및 조회
메서드 실행 전 캐시를 먼저 확인합니다. 데이터가 존재하면 결과를 즉시 반환하고, 없으면 메서드를 실행한 결과를 캐시에 저장합니다. 주로 읽기 작업이 많은 조회 서비스에 적용합니다.

```kotlin
@Cacheable(value = ["posts"], key = "#id")
fun getPostById(id: Long): PostResponse {
    // 캐시 미스 발생 시에만 실제 DB 로직이 실행됩니다.
    return postRepository.findByIdOrNull(id)?.toResponse() 
        ?: throw EntityNotFoundException()
}
```

#### @CachePut: 캐시 갱신
@Cacheable과 달리 메서드를 무조건 실행합니다. 실행 결과로 얻은 데이터를 캐시에 업데이트하며, 캐시 내용을 항상 최신 상태로 유지해야 하는 수정(Update) 기능에 활용합니다.

#### @CacheEvict: 캐시 삭제
캐시에서 특정 데이터를 제거합니다. 데이터가 삭제되거나 수정되어 기존 캐시가 더 이상 유효하지 않을 때 사용합니다. allEntries = true 옵션을 사용하여 특정 네임스페이스의 모든 데이터를 일괄 삭제할 수도 있습니다.

## 캐시 키 생성 로직과 중복 발생 원인

### 1. 기본 키 생성 전략(SimpleKeyGenerator)

Spring Cache는 별도의 키를 지정하지 않으면 DefaultKeyGenerator(정확히는 SimpleKeyGenerator)를 사용하여 키를 생성합니다. 이 생성기는 메서드의 파라미터들을 조합하여 키를 만듭니다.

| 파라미터 상태 | 생성되는 키의 형태 |
| :--- | :--- |
| 파라미터 없음 | SimpleKey.EMPTY |
| 파라미터 1개 | 해당 파라미터 값 자체 |
| 파라미터 여러 개 | 모든 파라미터를 포함한 SimpleKey 객체 |

### 2. 의도치 않은 키 중복의 메커니즘

복합적인 파라미터를 가진 메서드에서 명시적 키 설계가 필요한 이유는 다음과 같은 내부 동작 때문입니다.

* 객체의 동일성 판단: SimpleKey는 내부적으로 파라미터들의 hashCode()와 equals()를 이용합니다. 만약 파라미터로 넘겨받은 객체의 hashCode가 적절히 재정의되지 않았다면, 내용이 같아도 다른 키로 인식하거나 반대의 경우가 생길 수 있습니다.
* 네임스페이스 공유: 서로 다른 메서드임에도 불구하고 우연히 같은 타입과 순서의 파라미터를 가진다면, 의도치 않게 같은 캐시 영역(Cache Name) 내에서 키 충복이 발생하여 엉뚱한 데이터를 반환받을 위험이 있습니다.

## Redis 등 외부 저장소에서 직렬화가 필수적인 이유

### 1. 메모리 격리와 데이터의 전송

ConcurrentMap과 같은 로컬 캐시는 애플리케이션과 같은 JVM 메모리 공간을 공유합니다. 따라서 객체의 주소값만 가지고도 데이터를 참조할 수 있어 별도의 변환이 필요 없습니다.

하지만 Redis는 애플리케이션 외부에 존재하는 독립적인 프로세스입니다. 따라서 네트워크를 통해 데이터를 주고받아야 하며, 이 과정에서 객체를 전송 가능한 상태로 바꾸는 작업이 반드시 수반되어야 합니다.

### 2. 직렬화(Serialization)의 역할

직렬화는 메모리상의 객체(Object)를 바이트 스트림(Byte Stream)으로 변환하는 과정입니다.

* Serializable 인터페이스: 자바 시스템 간에 객체를 주고받을 때 최소한의 규격임을 선언하는 마커 인터페이스입니다. 이를 구현하지 않으면 자바 표준 직렬화 도구가 객체를 바이트로 쪼갤 수 없어 `NotSerializableException`이 발생합니다.
* JSON 직렬화: Redis 데스크탑 관리 도구로 데이터를 확인하거나, 다른 언어로 작성된 애플리케이션과 캐시를 공유해야 할 때는 자바 바이너리 데이터보다 JSON 형태가 훨씬 유리합니다. 이를 위해 Jackson 같은 라이브러리를 통해 직렬화 설정을 수행합니다.

| 구분 | 로컬 캐시 (ConcurrentMap) | 외부 캐시 (Redis) |
| :--- | :--- | :--- |
| 저장 위치 | 애플리케이션 내부 메모리 | 외부 원격 서버 메모리 |
| 전송 방식 | 참조(Reference) 전달 | 네트워크 통신 (I/O) |
| 필수 요건 | 없음 | 직렬화/역직렬화 로직 |

## 결론 및 실무 제안

결과적으로 명확한 캐시 키 설계는 데이터의 고유성을 보장하여 비즈니스 오류를 방지하기 위함이며, 직렬화 설정은 외부 저장소와의 통신을 위한 필수적인 데이터 변환 절차입니다.

실무에서는 다음과 같은 방식을 권장합니다.

* 키 설계: 가급적 `key = "#p0"` 또는 `key = "#id"`와 같이 비즈니스 핵심 식별자를 명시하여 키의 예측 가능성을 높여야 합니다.
* 저장 방식: 운영 환경에서는 확장성과 가독성을 고려하여 자바 기본 직렬화보다는 JSON 직렬화 방식을 설정하여 Redis에 저장하는 것을 제안합니다.

애플리케이션의 아키텍처가 단일 서버에서 분산 환경으로 넘어가는 시점이라면, 위에서 언급한 직렬화와 키 전략을 미리 점검하여 데이터 파편화 문제를 예방하시기 바랍니다.

혹시 현재 프로젝트에서 Redis를 도입할 때 객체 그래프가 복잡하여 직렬화 성능이 걱정되는 상황인가요? 상황에 따라 더 효율적인 직렬화 라이브러리를 선택하는 방법을 논의해 볼 수 있습니다.

## 캐시 저장소(Cache Manager)의 선택

추상화 덕분에 설정 변경만으로 저장소를 쉽게 교체할 수 있습니다. 서비스의 규모와 환경에 맞는 선택이 중요합니다.

| 저장소 종류 | 특징 | 추천 환경 |
| :--- | :--- | :--- |
| ConcurrentMap | 자바 Heap 메모리에 저장. 별도 설정 없이 즉시 사용 가능. | 로컬 개발 및 소규모 단일 서버 |
| Caffeine | 고성능 로컬 캐시 라이브러리. 만료 정책 및 크기 제한 설정 정교함. | 고성능이 필요한 단일 서버 환경 |
| Redis | 외부 분산 메모리 저장소. 서버가 재시작되어도 데이터 유지 가능. | 분산 환경 및 마이크로서비스(MSA) |
| Ehcache | 자바 전통의 강자. 로컬 및 분산 캐시를 모두 지원하며 디스크 저장 가능. | 대용량 데이터 및 다양한 캐시 전략 필요 시 |

## 실전 사용 시 고수의 팁과 주의사항

### 1. 프록시 내부 호출 문제 (Self-Invocation)
Spring Cache는 프록시 기반으로 동작하므로, 같은 클래스 내부에서 @Cacheable이 붙은 메서드를 직접 호출하면 캐시가 작동하지 않습니다. 프록시 객체를 거치지 않고 타겟 객체의 메서드를 직접 호출하기 때문입니다. 이 경우 별도의 서비스 클래스로 로직을 분리하는 것이 가장 권장되는 해결책입니다.

### 2. 캐시 키(Key) 설계와 직렬화
파라미터가 복합적일 경우 key = "#id" 처럼 SpEL(Spring Expression Language)을 사용하여 명확하게 지정해야 의도치 않은 키 중복을 방지할 수 있습니다. 또한 Redis와 같은 외부 저장소를 사용할 경우, 저장할 객체(DTO)는 반드시 Serializable 인터페이스를 구현하거나 JSON 직렬화 설정을 완료해야 합니다.

### 3. 설정 방법 (Quick Start)
구현을 위해서는 다음의 3단계를 수행합니다.

1. 의존성 추가: 빌드 도구(Gradle/Maven)에 spring-boot-starter-cache를 포함합니다.
2. 기능 활성화: 메인 설정 클래스에 @EnableCaching 어노테이션을 추가합니다.
3. 적용: 적절한 서비스 레이어 메서드 위에 캐시 어노테이션을 선언합니다.

## 결론

Spring Cache Abstraction은 비즈니스 로직과 기술적인 캐싱 처리를 분리하여 코드의 순수성을 유지하면서도 강력한 성능 최적화를 가능하게 합니다. 초기 단계에서는 단순한 ConcurrentMap으로 시작하되, 서비스 규모가 확장됨에 따라 Redis와 같은 분산 캐시 시스템으로 점진적으로 이전하는 전략을 제안합니다.

캐싱은 단순히 빠른 응답을 넘어 시스템 전체의 부하를 줄이는 핵심 전략임을 인지하고, 적재적소에 올바른 전략을 적용하시기 바랍니다. 현재 개발 중인 프로젝트가 단일 서버 환경인지 혹은 다수의 노드가 존재하는 분산 환경인지에 따라 최적의 CacheManager 설정은 달라질 수 있으므로 이를 반드시 먼저 확인해야 합니다.
