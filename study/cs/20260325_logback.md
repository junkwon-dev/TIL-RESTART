## 로깅의 필요성과 디버깅의 한계

소프트웨어 개발 과정에서 발생한 문제를 추적하기 위해 많은 개발자가 초기에는 단순한 출력문이나 디버거를 활용합니다. 하지만 실제 운영 환경에서는 사용자가 직접 시스템 내부의 동작을 확인할 수 없으며, 예기치 못한 에러가 발생했을 때 원인을 파악하기 위해 시스템 기록이 필수적입니다.

단순 출력문은 성능 저하를 유발하고 기록의 관리가 어렵다는 단점이 있습니다. 시스템 로그는 프로그램의 개발, 유지보수, 보안 감사 등 전반에 걸쳐 사용되며 운영의 안정성을 담보하는 핵심 요소입니다. 이 글에서는 자바 생태계에서 표준으로 자리 잡은 로깅 프레임워크의 개념과 활용 방법을 상세히 다룹니다.

---

## 로깅 프레임워크의 장점과 구조

로그(Log)는 기존의 디버거 방식에 비해 명확한 관리 포인트를 제공합니다. 단순 출력을 넘어선 로깅 시스템의 주요 이점은 다음과 같습니다.

* 상황별 레벨 지정: 메시지의 중요도에 따라 출력 여부를 선택할 수 있습니다.
* 흐름 및 에러 추적: 응용 프로그램의 실행 흐름을 시각화하고 에러 발생 지점을 정확히 포착합니다.
* 유연한 환경 조성: 프레임워크를 통해 복잡한 설정 없이도 일관된 로깅 환경을 구축합니다.
* 출력 최적화: 콘솔, 파일, 데이터베이스 등 다양한 위치로 로그를 전송하며 형식을 자유롭게 지정합니다.

### 로깅 라이브러리의 계층 구조

자바 진영에서는 SLF4J라는 추상화 계층을 중심으로 다양한 구현체가 존재합니다.

| 분류 | 종류 | 설명 |
| :--- | :--- | :--- |
| 인터페이스 (Facade) | SLF4J | 로깅 구현체에 의존하지 않도록 돕는 추상 레이어입니다. |
| 구현체 (Implementation) | Logback, Log4j2 | 실제 로깅 기능을 수행하며 속도와 메모리 효율을 담당합니다. |



SLF4J(Simple Logging Facade For Java)는 로거의 추상체 역할을 수행합니다. 이를 사용하면 내부 구현체인 Logback이나 Log4j2를 최소한의 설정 변경만으로 교체할 수 있는 유연성을 확보하게 됩니다.

## Logback의 핵심 개념과 로그 레벨

현재 스프링 부트의 기본 로깅 구현체는 Logback입니다. 과거에 널리 쓰이던 Log4j의 후속작으로, 더 빠른 처리 속도와 적은 메모리 점유율을 자랑합니다.

### 로그 레벨의 심각도

Logback은 5단계의 로그 레벨을 제공하며, 설정된 레벨 이상의 메시지만 출력하는 필터링 메커니즘을 가집니다.

1.  ERROR: 즉시 조치가 필요한 심각한 문제 발생 시 사용합니다.
2.  WARN: 당장 서비스 중단은 없으나 향후 잠재적 문제가 될 수 있는 주의 사항입니다.
3.  INFO: 운영 환경에서 참고할 만한 주요 비즈니스 프로세스의 완료 정보입니다.
4.  DEBUG: 개발 단계에서 변수 값 확인이나 SQL 로깅을 위해 사용합니다.
5.  TRACE: 가장 상세한 정보를 포함하며 모든 실행 흐름을 추적합니다.

심각도 수준은 다음과 같습니다.
> ERROR > WARN > INFO > DEBUG > TRACE

## Logback 설정

스프링 부트에서는 logback-spring.xml 파일을 통해 상세한 로깅 정책을 수립할 수 있습니다. 주요 구성 요소인 Appender, Encoder, Logger의 관계를 이해하는 것이 중요합니다.

### 주요 구성 요소

* Appender: 로그를 출력할 대상을 결정합니다. (Console, File, SMTP 등)
* Encoder: 로그 메시지를 지정된 패턴에 맞춰 변환합니다.
* Logger: 특정 패키지별로 로그 레벨을 설정하고 Appender에 전달합니다.

### Appender 종류 및 속성

| 명칭 | 출력 대상 | 특징 |
| :--- | :--- | :--- |
| ConsoleAppender | 콘솔창 | 개발 단계에서 실시간 확인용으로 적합합니다. |
| RollingFileAppender | 파일 | 용량이나 날짜에 따라 파일을 분리하여 저장합니다. |
| SMTPAppender | 이메일 | 에러 발생 시 관리자에게 즉시 메일을 발송합니다. |

### 로그 패턴 정의

Encoder 내부에 작성하는 패턴 문자열은 로그의 가독성을 결정합니다.

* %d: 로그 기록 시간 (예: %d{yyyy-MM-dd HH:mm:ss})
* %p: 로깅 레벨
* %t: 현재 스레드 명
* %m: 로그 메시지 본문
* %n: 줄바꿈

## 설정 예시: 비동기 및 롤링 정책

실무에서는 대량의 로그로 인한 성능 저하를 막기 위해 비동기 로깅(AsyncAppender)과 파일 관리 정책을 적용합니다. 다음은 실무에서 활용되는 logback-spring.xml의 구성 예시입니다.

```xml
<included>
    <include resource="org/springframework/boot/logging/logback/defaults.xml"/>
    <property name="LOG_FILE_BASE" value="myapp"/>

    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>log/${LOG_FILE_BASE}.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>log/${LOG_FILE_BASE}.log.%d{yyyyMMdd}.%i</fileNamePattern>
            <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                <maxFileSize>500MB</maxFileSize>
            </timeBasedFileNamingAndTriggeringPolicy>
            <maxHistory>60</maxHistory>
        </rollingPolicy>
        <encoder>
            <pattern>${FILE_LOG_PATTERN}</pattern>
        </encoder>
    </appender>

    <appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
        <queueSize>1024</queueSize>
        <appender-ref ref="FILE"/>
    </appender>

    <root level="INFO">
        <appender-ref ref="ASYNC_FILE"/>
    </root>
</included>
```

위 설정은 로그 파일이 500MB를 초과하거나 날짜가 변경되면 새로운 파일을 생성하고, 최대 60일간 보관하도록 지정합니다. 또한 AsyncAppender를 사용하여 로깅 작업이 메인 비즈니스 로직의 성능에 미치는 영향을 최소화합니다.

## 실전 추가 학습

### 실전 추가 학습 1. AsyncAppender: 로깅 성능 최적화의 핵심

기본적인 `Appender`들은 로그 기록 방식이 동기(Synchronous)로 작동합니다. 즉, 로그가 파일이나 네트워크에 쓰여질 때까지 애플리케이션의 메인 비즈니스 로직(Thread)이 잠시 멈추게 됩니다. 대규모 트래픽 환경에서는 이 찰나의 대기가 병목 현상을 유발합니다.

* **작동 원리**: `AsyncAppender`는 내부에 **BlockingQueue**를 두고, 로그 이벤트를 큐에 쌓은 뒤 별도의 워커 스레드가 이를 실제 Appender(File, Console 등)로 전달합니다.
* **주요 설정 옵션**:
    * `queueSize`: 큐의 크기 (기본값 256). 메모리 상황에 맞게 조절합니다.
    * `discardingThreshold`: 큐가 차오를 때 TACE, DEBUG, INFO 로그를 버릴지 결정하는 임계값입니다. (기본값 20% 남았을 때 삭제 시작) 이를 `0`으로 설정하면 로그 유실 없이 모든 로그를 유지합니다.
    * `includeCallerData`: 호출자의 클래스명, 라인 번호 등의 정보를 포함할지 여부입니다. `true`로 설정 시 성능 저하가 발생할 수 있어 주의가 필요합니다.



---

### 실전 추가 학습 2. 쿠버네티스(Kubernetes) 환경에서의 로깅 전략

클라우드 네이티브 및 쿠버네티스 환경에서는 로그를 파일로 저장하고 관리하는 전통적인 방식(`RollingFileAppender`)보다 **표준 출력(STDOUT)** 방식을 권장합니다.

* **Stateless 원칙**: 컨테이너는 언제든 생성되고 삭제될 수 있습니다. 로그를 컨테이너 내부 파일로 저장하면 컨테이너 삭제 시 로그도 사라집니다.
* **표준 출력(Standard Out)**: 애플리케이션은 단순히 `ConsoleAppender`를 통해 로그를 표준 출력으로 내보냅니다.
* **Log Collector(Sidecar/DaemonSet)**: 쿠버네티스 노드에 상주하는 **Fluentbit**나 **Filebeat** 같은 수집기가 이 표준 출력을 가로채어 Elasticsearch, CloudWatch, Datadog 등으로 전송합니다.
* **권장 사항**: 쿠버네티스에서는 로그 파일을 직접 관리하기보다, 구조화된 로그(JSON)를 표준 출력으로 던지고 외부 로깅 플랫폼(EFK Stack 등)에서 통합 관리하는 것이 정석입니다.

### 실전 추가 학습 3. 구조화된 로깅을 위한 LogstashEncoder

로그를 단순 텍스트(`PatternLayout`)로 남기면 사람이 읽기는 좋지만, 기계(Logstash, Elasticsearch)가 분석하기는 어렵습니다. 검색과 필터링 효율을 극대화하기 위해 로그를 **JSON 형식**으로 변환해주는 `LogstashEncoder`가 널리 사용됩니다.

### LogstashEncoder의 장점
* **필드 자동 분리**: 시간, 레벨, 메시지, 스레드 등이 각각의 JSON 키값으로 분리되어 저장됩니다.
* **MDC(Mapped Diagnostic Context) 연동**: 사용자 ID나 트랜잭션 ID 등을 로그에 심었을 때, 별도 설정 없이 JSON 필드로 자동 변환됩니다.
* **확장성**: 사용자 정의 필드를 쉽게 추가할 수 있어 서비스 모니터링 시 매우 강력합니다.

### 설정 예시 (logback-spring.xml)
```xml
<appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <includeMdc>true</includeMdc>
    </encoder>
</appender>
```
위와 같이 설정하면 로그가 아래와 같은 JSON 형태로 출력되어 검색 엔진에서 즉시 인덱싱될 수 있습니다.
```json
{"@timestamp":"2026-03-25T21:46:02.123+09:00","level":"INFO","thread_name":"http-nio-8080-exec-1","logger_name":"com.myapp.Service","message":"Order processed","orderId":"ORD-123"}
```

### 요약: 현대적인 로깅 스택 구성

| 구분 | 추천 방식 | 이유 |
| :--- | :--- | :--- |
| **인터페이스** | SLF4J | 구현체 교체 용이성 (유연성) |
| **구현체** | Logback / Log4j2 | 성능 및 기능적 완성도 |
| **처리 방식** | **AsyncAppender** | 비즈니스 로직 성능 보호 (Non-blocking) |
| **포맷** | **LogstashEncoder** (JSON) | 로그 분석 및 검색 효율화 (구조화) |
| **출력/수집** | **Console(STDOUT)** + 수집기 | 쿠버네티스 환경 최적화 및 영속성 확보 |

## 결론

효율적인 로깅 시스템은 단순한 기록을 넘어 시스템의 생존 여부를 판단하고 성능을 최적화하는 나침반 역할을 수행합니다. SLF4J와 Logback을 조합한 표준 방식을 준수함으로써 유연하고 강력한 모니터링 기반을 마련할 수 있습니다.
